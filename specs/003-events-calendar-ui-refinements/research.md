# Phase 0 Research: Events, Calendar & Navigation UI Refinements

No `NEEDS CLARIFICATION` markers remain in the Technical Context. The decisions below are about *how* to refactor already-shipped code, not new technology choices — there is no new dependency or backend surface to evaluate.

## 1. Full management parity on slug-viewed trees requires no new authorization

- **Decision**: Verified against `supabase/migrations/0007_rls_policies.sql` that `individuals_admin_editor_insert/update/delete` and `relationships_admin_editor_insert/delete` scope write access by `profiles.role`, not by whether the target row's `family_tree_id` belongs to the tree currently marked `is_default`. The same holds for `family_trees_admin_update`/`_delete`/`_insert` (Admin, any tree). Conclusion: **no new migration or RLS policy is needed** for spec FR-002 — an Admin/Editor already has full backend permission to manage any tree; the only thing missing today is a frontend UI to use that permission from a slug-viewed tree.
- **Rationale**: This is the single most important finding for this feature's risk profile — it means "full management parity" is a pure frontend change with zero new attack surface, not a permissions expansion that would need security review. It also confirms the spec's Clarification answer is actually cheap to build, which is why it was accepted as the resolution rather than the narrower "viewing options only" alternative.
- **Alternatives considered**: Adding a new RLS check scoped to "any tree the caller has rights to" — rejected as unnecessary; the existing role-only check already is that.

## 2. Extracting a shared `TreeWorkspace` component

- **Decision**: Move `Home.tsx`'s sidebar (all three sections), `TreeCanvas`, `IndividualDetailPanel`, and all five modals (create/edit/delete individual, create relationship, import) into a new `src/features/tree/TreeWorkspace.tsx`, parameterized by `treeId`, `treeName` (sidebar title), and `upcomingEventsPath` (so the sidebar's "Sự kiện sắp tới" link points at the right route for whichever tree is open). `Home.tsx` and `TreeBySlug.tsx` keep their own tree-resolution logic (default vs. slug) and their own loading/not-found/guest messaging — which genuinely differ — and become thin wrappers that render `<TreeWorkspace>` once a tree is resolved.
- **Rationale**: Spec FR-002 requires *identical* capability regardless of how a tree was reached. Duplicating ~200 lines of modal/CRUD wiring across two page components would immediately drift (a fix or new sidebar item added to one would silently not apply to the other) — exactly the kind of bug this feature exists to close. A single shared component with the resolution logic kept separate (since that part is legitimately different between the two entry points) is the standard React pattern for "same capability, different data source."
- **Alternatives considered**: A React Context/provider wrapping both routes — rejected as more indirection than needed for two call sites; a plain shared component with props is simpler and easier to test in isolation. Keeping the duplication and manually mirroring every future change — rejected outright as the source of the very bug being fixed.

## 3. One `UpcomingEvents` page, two routes

- **Decision**: Add `/:slug/su-kien-sap-toi` as a second route to the existing `UpcomingEvents` page component (declared *before* the catch-all `/:slug` route in `router.tsx`, for readability/safety even though React Router's own ranking already resolves this correctly). Inside the component, an optional `slug` route param determines tree resolution: present → `getFamilyTreeBySlug(slug)`; absent → `getDefaultFamilyTree()` (today's behavior, unchanged for the home path).
- **Rationale**: The calendar UI, derivation hook, and popup logic (User Stories 2–3) are identical regardless of which tree is being viewed — only *which tree's individuals* feed into `useMonthEvents` differs. One component with a resolution branch avoids duplicating the calendar UI itself, mirroring the same "shared UI, different data source" shape as `TreeWorkspace`.
- **Alternatives considered**: A separate `UpcomingEventsBySlug.tsx` page — rejected, would duplicate the entire calendar/popup implementation for a one-line difference in tree resolution.

## 4. Reusing `Modal` for the date-detail popup

- **Decision**: Wrap the existing `DayEventsPanel` in the app's existing generic `Modal` component (`src/app/Modal.tsx`, already used for every create/edit/delete/import dialog) instead of building a new popup/overlay primitive. The Modal's own title bar carries the selected date's heading (Gregorian + inline lunar), and `DayEventsPanel`'s content becomes just the event list or "no events" message.
- **Rationale**: `Modal` already implements the exact behavior spec FR-005/FR-006 need (overlay, explicit close, returns to the underlying view with nothing else obscuring it) and is the app's one established popup pattern — introducing a second, bespoke popup component for this one case would be an unjustified new pattern for no functional benefit.
- **Alternatives considered**: A custom inline dropdown/tooltip anchored to the clicked day cell — rejected; the spec explicitly asks for a "popup," and a modal overlay is simpler to get right for keyboard/click-outside dismissal than a positioned tooltip, consistent with how every other "click something, see detail" interaction in this app already works.

## 5. Inline lunar-date format

- **Decision**: `formatLunarDate` changes from the current Vietnamese-prose sentence ("Ngày 10 tháng 12 năm 2024 (âm lịch)") to `DD/MM/YYYY AL` (e.g. `10/12/2024 AL`), with a trailing `" (nhuận)"` appended only when `isLeapMonth` is true (e.g. `20/04/2020 AL (nhuận)`). Callers pair it inline with the existing Gregorian `DD/MM/YYYY` text (already produced by `formatPartialDate` for full-precision dates) as `"<gregorian> (<lunar>)"`, e.g. `01/01/2025 (10/12/2024 AL)` — matching the request's own example exactly.
- **Rationale**: `AL` (Âm Lịch) as a compact suffix, paired with the same `DD/MM/YYYY` shape already used for the Gregorian side, is the most direct reading of the requested example and keeps both dates visually parallel. The leap-month qualifier is kept as a short trailing note rather than folded unreadably into the numeric month (e.g. no "04N"), preserving spec FR-008's "unambiguous" requirement without breaking the base pattern for the common (non-leap) case.
- **Alternatives considered**: Encoding the leap flag into the month number itself (e.g. `04*`) — rejected as more cryptic than a plain word; dropping the leap indicator entirely — rejected outright, spec FR-008 explicitly forbids silently dropping it. Applying the same full format inside the calendar grid's per-day cells — rejected (see spec Assumptions): a 7-column grid cell has no room for a full `DD/MM/YYYY AL` string; those cells keep their existing compact `d/m` + leap-flag indicator, unchanged by this feature.
