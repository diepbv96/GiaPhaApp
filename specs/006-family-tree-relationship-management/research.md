# Research: Family Tree Naming, Multi-Tree Membership & Relationship Management

## 1. Renaming a family tree (FR-001–FR-003)

**Finding**: `family_trees` already has `name text not null check (char_length(trim(name)) > 0)` (`0003_family_trees.sql`) and RLS already includes `family_trees_admin_update` — an admin-only, unrestricted `UPDATE` policy (`0007_rls_policies.sql:44-47`), not scoped to any particular column. `treeService.ts` already has an analogous `updateTreeSlug(treeId, slug)` function and `TreeManagement.tsx`/`SlugField.tsx` already implement the "open a modal, edit one field, save" pattern for `slug`. There is currently no `updateFamilyTreeName()` function and no "Sửa tên" UI action.

**Decision**: Add `updateFamilyTreeName(treeId: string, name: string): Promise<FamilyTreeSummary>` to `treeService.ts`, copying `updateTreeSlug`'s error-mapping shape: `23514` (the existing non-empty-name `CHECK` constraint) → `VALIDATION_FAILED` ("Tên không được để trống."), `42501` → `PERMISSION_DENIED` ("Chỉ quản trị viên mới có thể sửa tên cây gia phả."). Add a `TreeNameField.tsx` component mirroring `SlugField.tsx` (trim-on-submit, disable Save when unchanged/empty) and wire it into `TreeManagement.tsx` next to the existing "Sửa slug" action. No new migration needed — the DB already permits this; only the client was missing.

**Propagation check**: Every screen that renders a tree's name (`Home.tsx`, `TreeBySlug.tsx`, `TreeManagement.tsx`) sources it from a `useQuery` keyed under the shared `["family-trees", ...]` prefix, and every existing tree mutation already calls `queryClient.invalidateQueries({ queryKey: ["family-trees"] })` (a prefix match invalidates all of them). `updateFamilyTreeName` following the same pattern satisfies SC-001 with no additional wiring.

**Alternatives considered**:
- A dedicated rename-only RPC function (like `set_default_family_tree`): rejected — a single-column `UPDATE` under an existing unconditional admin policy needs no transaction-safety guarantee beyond what a plain `UPDATE` already gives (unlike the "exactly one default" invariant `set_default_family_tree` protects).

## 2. Multi-tree membership: schema shape (FR-004–FR-008, FR-016)

**Finding**: `individuals.family_tree_id` is `not null references family_trees(id) on delete cascade` (`0004_individuals.sql`) — a single, required FK. It is read on write (`individualService.ts` `createIndividual(treeId, ...)`) and on every "which people are in this tree" query (`treeGraphService.ts`'s `getTreeGraph`, `importService.ts`'s duplicate-check query), but grep across `src/` shows `Individual.familyTreeId` (the mapped camelCase field) is **never read** anywhere for logic — it's written by `mapIndividualRow` and never consulted again. This means the column's *storage* is load-bearing (several queries filter by it, and it can't be null) but its *semantic meaning* is not hard-coded into any business rule the frontend enforces.

**Decision**: Introduce a new join table, `individual_tree_memberships (individual_id, family_tree_id, created_at)`, `primary key (individual_id, family_tree_id)`, both columns `references ... on delete cascade`, as the sole authority for "which family trees is this person a member of." Keep `individuals.family_tree_id` exactly as-is (still `not null`, still set at creation) — it now means "the tree this person was originally created in," a historical fact that seeds their first membership row via a new `AFTER INSERT` trigger (`seed_individual_primary_tree_membership`), and is otherwise not treated as special (removing it as a membership later, per Edge Cases, is allowed like any other membership as long as at least one remains — FR-007). A one-time backfill (`insert into individual_tree_memberships select id, family_tree_id, created_at from individuals`) populates the table for every pre-existing individual in the same migration.

**Query-side change**: `getTreeGraph(treeId)`'s individuals query moves from `supabase.from("individuals").select(...).eq("family_tree_id", treeId)` to a PostgREST embedded-resource filter: `supabase.from("individuals").select(\`${INDIVIDUAL_COLUMNS}, individual_tree_memberships!inner(family_tree_id)\`).eq("individual_tree_memberships.family_tree_id", treeId)` — one round trip, same shape returned to callers (the embedded `individual_tree_memberships` field is not part of `Individual` and is dropped by `mapIndividualRow`, which only reads the columns it already knows about).

**FR-013 (relationship trigger)**: `enforce_relationship_same_tree()` (`0005_relationships.sql`) currently resolves `a_tree`/`b_tree` via `select family_tree_id from individuals where id = ...`. It's rewritten to check membership instead: `exists (select 1 from individual_tree_memberships where individual_id = new.person_a_id and family_tree_id = new.family_tree_id)` (and the same for `person_b_id`) — this is what actually makes "person belongs to more than one tree, but a relationship in Tree A must still only link Tree-A members" hold at the database level, not just in the UI's selection list.

**FR-007 (can't remove last membership)**: Enforced by a `BEFORE DELETE` trigger on `individual_tree_memberships` (`enforce_last_tree_membership`) that raises if the row being deleted is the individual's only remaining membership — the same "trigger/RLS is the authoritative enforcement, frontend is UX-only" convention used everywhere else in this schema (`0007_rls_policies.sql` header comment, `family_trees_limit_check`, `family_trees_single_default`).

**Alternatives considered**:
- Dropping `individuals.family_tree_id` and making the join table the only place a tree relationship is recorded: rejected — it's a required, indexed, `NOT NULL` column referenced by name in `individualService.ts` (insert payload), `importService.ts` (duplicate-check query and row insert), and the mapper; removing it would touch every one of those call sites for a rename-only semantic gain that FR-004–FR-008 don't actually require (they only require that membership be *addable/removable*, not that there be no "first" tree).
- A `is_primary boolean` flag on the membership rows instead of keeping the separate `individuals.family_tree_id` column: rejected — adds a second "exactly one primary per individual" invariant to maintain (mirroring the `family_trees_single_default` pattern) for a distinction nothing in the spec's functional requirements or success criteria depends on; the existing column already fills that role for free.
- Letting the frontend compute "which individuals are in this tree" by fetching all memberships separately and filtering client-side: rejected — `getTreeGraph` is called on every tree view including as a guest/`anon` viewer of a public tree (`0009_public_tree_access.sql`), and keeping it a single filtered query (rather than "fetch everything, filter in JS") avoids ever transmitting an individual's *other*-tree membership existence to the client at all, which is also how FR-017 ends up correct by construction rather than by a client-side redaction step.

## 3. Adding/removing a person's tree membership: UI shape (FR-005, FR-006, User Story 2)

**Finding**: Every existing per-person action (`Sửa thông tin`, `Thêm mối quan hệ`, `Xoá cá thể`) in `TreeWorkspace.tsx` starts from an already-selected individual within the current tree and opens a modal — there's no existing "search people across all trees" UI anywhere in the app (import's Excel flow is the only bulk cross-tree-adjacent feature, and it creates new rows, not links to existing ones).

**Decision**: Model "add to another tree" the same way — an action on the already-selected individual, not a new global person-search screen. The action opens a small dialog (`ManageTreeMembershipDialog.tsx`) listing `getFamilyTrees()` results the person is **not** already a member of (cross-referenced against `getIndividualTreeMemberships(individual.id)`) as the "add to" choices, and their current memberships (each with a "remove" button, disabled/hidden when it's their only one, matching FR-007) below that. This matches every one of spec.md's User Story 2 acceptance scenarios, which are all phrased "a person exists ... add that person to Family Tree B" (person-first, destination-tree-second) — none of them describe searching for a person from a tree-membership-management screen.

**Alternatives considered**:
- A cross-tree person search-and-import screen (search by name across every tree, then pick a destination): rejected as unnecessary complexity for a 5-tree-max app where the natural entry point is always "I'm looking at this specific person, and I want them in one more tree" — no spec scenario requires discovering a person you don't already have selected.

## 4. Managing a person's relationships: what's actually missing (FR-009–FR-014, User Story 1)

**Finding**: `RelationshipForm.tsx` already creates relationships, and its `individuals` prop is always fed `graph.individuals` from `getTreeGraph(treeId)` (`TreeWorkspace.tsx:226`, `IndividualForm.tsx`'s "Liên kết với" picker) — so FR-012/FR-013 ("selection list must display persons in same family tree only") is **already satisfied today**, and stays satisfied after §2's `getTreeGraph` change, with no new code. What's actually missing: `relationshipService.ts` exports `deleteRelationship(id)` but grep shows **zero call sites** — no button anywhere calls it. There is **no `updateRelationship`** function at all, and `relationships`' RLS policies (`0007_rls_policies.sql:88-96`) only cover `select`/`insert`/`delete` — no `update` policy exists, so even a client-side `.update()` call would silently affect 0 rows. `IndividualDetailPanel.tsx`'s "Gia đình" section (`PersonList`) only ever renders resolved *people*, grouped by relation kind (`getFamilyRelations`) — never the underlying `Relationship` row's `id`/`type`, so there's currently no way for a click handler to know *which* relationship row to edit or delete.

**Decision**:
1. Add `updateRelationship(id: string, type: RelationshipType): Promise<Relationship>` to `relationshipService.ts`, reusing `createRelationship`'s duplicate-check-then-write shape (the existing `relationships_unique_edge` constraint `unique (type, person_a_id, person_b_id)` already rejects a same-tree duplicate on `UPDATE` too, since Postgres `UNIQUE` constraints apply to both statement types; map its `23505` the same way `createRelationship` does).
2. Add migration `0018_relationships_admin_editor_update.sql`: an `UPDATE` RLS policy for `admin`/`editor`, copy-pasted in structure from `relationships_admin_editor_delete` (same role check, same `to authenticated`).
3. In `IndividualDetailPanel.tsx`, resolve the underlying `Relationship` row per listed relative (by matching `personAId`/`personBId` against the selected individual and the listed person's id in `graph.relationships`) so an edit/delete control can be attached per row when `canManage` is true — this is the one genuinely new piece of UI logic, since today's `PersonList` only ever receives already-resolved `Individual[]` with no relationship-id trail back to its source edge.

**Alternatives considered**:
- Wiring up the existing unused `deleteRelationship` as the only change, and treating "update" as "delete + re-create": rejected — re-creating loses the relationship's `id` (irrelevant here, since nothing references a relationship by id elsewhere) but more importantly means two round trips and two possible partial-failure states (delete succeeds, create fails) for what should be a single atomic type change; a real `UPDATE` avoids that.

## 5. Public-tree membership privacy (FR-017, SC-006 — from `/speckit-clarify`)

**Finding**: `0009_public_tree_access.sql` grants `anon` `SELECT` on `family_trees`/`individuals`/`relationships`, each scoped with `exists (select 1 from family_trees where ... and family_trees.is_public)` — i.e., anon access is opt-in per table, added only where a guest-facing feature needs it, with no blanket "anon can read everything a signed-in viewer can."

**Decision (superseded — see below)**: Add no `anon` policy at all for `individual_tree_memberships`. Postgres RLS defaults to deny with no matching policy, so an anonymous guest's query against that table returns zero rows regardless of what they ask for — satisfying FR-017/SC-006 by omission, the same way `import_batches`/`import_row_results` (which also have no `anon` policy) are already invisible to guests today. `SELECT` for authenticated users (any role, including `viewer` — FR-015) mirrors `individuals_select`'s `using (public.has_profile())`. `INSERT`/`DELETE` (adding/removing membership) are `admin`/`editor`-only, mirroring `individuals_admin_editor_insert`/`_delete`.

**Correction (post-deployment, migration `0019_public_tree_membership_visibility.sql`)**: the "no anon policy at all" decision above was wrong — it satisfies FR-017 but breaks guest viewing outright, because `getTreeGraph()`'s `individual_tree_memberships!inner(...)` embedded join (contracts/tree-membership.md) needs `anon` to read *some* rows here to see a public tree's own members at all, and `individuals_public_select` (`0009_public_tree_access.sql`) still only checked an individual's original `family_tree_id`, not their full membership set. Fixed by adding a public-tree-scoped `anon` `SELECT` policy on `individual_tree_memberships` (`using (exists (select 1 from family_trees where family_trees.id = individual_tree_memberships.family_tree_id and family_trees.is_public))`) and rewriting `individuals_public_select` to check membership instead of the individual's original tree. FR-017 still holds: `anon` can only ever read a membership row whose *own* tree is public, so a private-tree membership for the same person is still never exposed — the fix only widens which *public*-tree memberships are visible, exactly mirroring the existing `individuals_public_select`/`relationships_public_select` pattern.

## 6. Everything else — no new unknowns

- **Removing a person from a tree that still has relationships there (Edge Cases)**: reuses the exact confirmation-checkbox pattern `DeleteIndividualDialog`/`deleteIndividual` already established for whole-person deletion (spec feature 005's `contracts/delete-individual.md`) — `removeIndividualFromTree(individualId, treeId, { cascadeRelationships })` deletes that tree's relationships for the person first when confirmed, then the membership row. No new interaction pattern to design.
- **Import (`.xlsx`) duplicate-check and row-insert still key off `individuals.family_tree_id` directly** (`importService.ts`): left unchanged. Bulk import is not one of this feature's user stories, and per §2 that column remains a valid, populated field (the tree a batch-imported person is created in) — this feature does not change import behavior, and a person added to a *second* tree via this feature's new UI simply isn't reachable by that tree's import duplicate-check, which is an existing, documented-here, out-of-scope limitation rather than a regression.
- **Tree-rename and relationship-update input validation**: both reuse existing DB-level `CHECK` constraints (`family_trees.name`, relationship type enum) — no new client-side validation library/pattern needed beyond what `SlugField.tsx`/`RelationshipForm.tsx` already do.
