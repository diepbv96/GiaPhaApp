# Research: Individuals Admin Dashboard

## 1. Cross-tree, paginated individuals list (FR-001, FR-002, FR-014, SC-001, SC-005)

**Finding**: No existing service function queries `individuals` across all trees — `individualService.ts` has only tree-scoped or single-id functions, and `treeGraphService.ts`'s `getTreeGraph(treeId)` is always scoped to one tree. No query anywhere in the codebase uses `.range()`/`.limit()`; every existing list is a single unbounded fetch, viable today only because the app caps at 5 family trees with correspondingly small per-tree counts. Spec 007 explicitly targets 1,000+ individuals system-wide, breaking that assumption for the first time.

**Decision**: Add `listIndividualsAdmin(params: { page: number; pageSize: number; treeId?: string; search?: string }): Promise<{ individuals: IndividualWithTrees[]; total: number }>` to `individualService.ts`. Two-query shape per page:
1. **Selection query** — `supabase.from("individuals").select(INDIVIDUAL_COLUMNS, { count: "exact" })`, optionally `.eq("individual_tree_memberships.family_tree_id", treeId)` via an `!inner` embed when `treeId` is set (mirrors `treeGraphService.ts`'s existing pattern), optionally `.or(...)` search filter (§2), then `.range(page * pageSize, page * pageSize + pageSize - 1)` ordered by `full_name` for stable pagination. Supabase's `{ count: "exact" }` option returns the total matching row count alongside the page, needed for SC-001/pagination controls without a second round trip.
2. **Membership-display query** — a second query, `individual_tree_memberships` joined to `family_trees`, filtered `.in("individual_id", <ids from query 1>)`, with **no** `!inner`/tree filter, so it returns *every* tree each of this page's individuals belongs to (needed for FR-002's "show which family tree(s) they belong to," which the `!inner`-filtered query alone cannot provide once a tree filter is active — that filter only proves membership in the selected tree, not the person's other memberships).

**Rationale for two queries over one**: PostgREST's `!inner` embed, when combined with `.eq()` on the embedded column, filters the embedded array itself down to just the matching row(s) — it cannot simultaneously "filter individuals by tree X" and "show all trees per individual" in one embedded select. Two small, indexed queries (one page of ids, then a fan-out by those ids) is simpler and cheaper than fetching the full unfiltered join and re-deriving both memberships from it client-side.

**Alternatives considered**:
- A single query with a non-`!inner` (left) embed of `individual_tree_memberships` plus a separate `.filter()` on a computed "exists in tree X" condition: rejected — PostgREST has no server-side way to filter rows by "has an embedded child matching X" without `!inner`, so this would require fetching all rows and filtering client-side, defeating pagination.
- A dedicated Postgres view or RPC function pre-joining individuals + memberships + trees: rejected as unnecessary — two plain PostgREST queries against existing tables/indexes are enough at this scale (1,000s, not millions, of rows) and keep this feature's only new SQL confined to the search columns and one trigger fix (§4), matching the "RLS/triggers are authoritative, queries are just queries" convention already established.

## 2. Diacritic-tolerant name/alias search (FR-004, Assumption "tolerant of Vietnamese diacritics")

**Finding**: The Postgres `unaccent` extension is already installed (`0013_family_tree_slug.sql:5`) and already wrapped in a reusable normalization idiom for tree names: `extensions.unaccent(replace(lower(input), 'đ', 'd'))` (lowercase → `đ`→`d` first, since `unaccent` doesn't touch `đ`, a distinct Latin letter → then strip remaining combining diacritics). However, Supabase-JS's `.ilike()` (and every other PostgREST filter operator) only compares a **column** against a pattern — it cannot wrap a column in an arbitrary SQL function call inline, so this normalization expression cannot be used directly inside a client-issued filter.

**Decision**: Add two **generated, stored** columns to `individuals` in the new migration: `full_name_normalized text generated always as (public.normalize_search_text(full_name)) stored` and `alias_normalized text generated always as (public.normalize_search_text(alias)) stored`, each with a supporting `btree` index (or `gin`/`trgm` if substring performance ever needs it — not needed at 1,000-row scale, plain `btree` + `ilike '%term%'` is adequate here since this app has no full-text-search requirement beyond "contains"). The search term is normalized the same way **client-side** before the query — a new `normalizeSearchTerm()` in `src/features/individuals/individualSearch.ts`, reusing the exact lowercase → `đ`→`d` → NFD-strip-combining-marks technique already implemented in `src/lib/slug.ts`'s `slugify()` (just without the hyphenation/whitespace-collapsing steps, since search terms keep their spaces). `listIndividualsAdmin`'s search filter becomes `.or(\`full_name_normalized.ilike.%${term}%,alias_normalized.ilike.%${term}%\`)`.

**Correction (post-implementation)**: the decision above originally called `extensions.unaccent(replace(lower(...), 'đ', 'd'))` directly inline in the generated column expression, mirroring `slugify_tree_name()`'s function body verbatim. That fails at migration time with `42P17: generation expression is not immutable` — `unaccent()` is declared `STABLE`, not `IMMUTABLE`, and while `0013_family_tree_slug.sql`'s `slugify_tree_name()` gets away with calling it because Postgres trusts an explicit `immutable` function declaration without re-checking what it calls internally, a generated column's expression has no such wrapper to extend that trust to. Fixed by introducing `public.normalize_search_text(input text) returns text language sql immutable as $$ select extensions.unaccent(replace(lower(coalesce(input, '')), 'đ', 'd')); $$` and using *that* in both generated columns — same normalization, now with a trusted immutable wrapper. This function is also a natural candidate to eventually consolidate `slugify_tree_name()` onto, though this feature didn't touch that (out of scope).

**Alternatives considered**:
- A Postgres RPC function (`search_individuals(term text, ...)`) that applies `unaccent()` server-side per call: rejected — adds a bespoke function to maintain (with its own RLS/`grant execute` surface) for something a generated column already solves more simply and transparently, and generated columns stay queryable via plain PostgREST filters (reusable by any future feature) rather than being locked behind one RPC's parameter shape.
- Client-side filtering (fetch all individuals, `normalizeSearchTerm()` + `.includes()` in JS): rejected outright — defeats pagination and the 1,000+-row scale target (SC-001/SC-005) by requiring a full unbounded fetch on every keystroke.

## 3. Search input responsiveness (FR-004, Edge Case "large number of records")

**Finding**: No debounce utility exists anywhere in this codebase (`package.json` has no debounce dependency; no custom hook found).

**Decision**: Add a small local `useDebouncedValue<T>(value: T, delayMs: number): T` hook (`src/lib/useDebouncedValue.ts`, ~10 lines, a single `useEffect`+`setTimeout`) rather than adding a dependency. The dashboard page debounces the raw search-input state (e.g. 300ms) before it feeds `listIndividualsAdmin`'s `search` param / the `useQuery` key, so a paginated, indexed-but-still-network-bound query isn't re-issued on every keystroke.

**Alternatives considered**: `use-debounce`/`lodash.debounce` npm packages — rejected per the "zero new npm dependencies" constraint; a 10-line hook is simpler than a dependency for one call site.

## 4. Delete "everywhere": what already works, and the one real gap (FR-010, FR-011, Edge Cases)

**Finding**: `deleteIndividual(id, { cascadeRelationships })` (`individualService.ts:73-106`) already performs a **system-wide** delete matching spec 007's "full delete everywhere" requirement almost exactly as-is: its relationship-cleanup step (`.or(\`person_a_id.eq.${id},person_b_id.eq.${id}\`)`) has no tree filter, so it already removes every relationship referencing the person in *every* tree, not just one — and deleting the `individuals` row then cascades (`ON DELETE CASCADE`) to remove every `individual_tree_memberships` row for that person via the existing FK.

**The gap**: `individual_tree_memberships` carries two `BEFORE DELETE` guard triggers from spec 006 (`0017_individual_tree_memberships.sql`) — `enforce_last_tree_membership` (blocks deleting a person's *only* remaining membership row) and `enforce_no_relationships_before_membership_removal` (blocks deleting a membership row while relationships still reference that person in that tree). Both were designed for the *single-tree-removal* flow (`treeMembershipService.removeIndividualFromTree`, where the individual keeps existing in other trees), not for "the individual row itself is being deleted in the same transaction." For a person who belongs to **more than one** tree — exactly the case spec 006 introduced and spec 007's Edge Cases call out — deleting their `individuals` row cascades into deleting all N of their `individual_tree_memberships` rows within the same statement; each row's `BEFORE DELETE` trigger fires with Postgres's per-row command-counter visibility already reflecting earlier same-statement deletions, so by the last row, `enforce_last_tree_membership`'s `count(*) <= 1` check can see a count that has already dropped to 1 and incorrectly raise `LAST_TREE_MEMBERSHIP`, aborting the entire delete. (The second trigger is not at risk: `cascadeRelationships` already deletes *all* of the person's relationships, system-wide, before the `individuals` row delete runs, so by the time the membership cascade fires, no relationship rows remain to trip it.) This exact interaction has never been exercised by any test in the repo.

**Decision**: In the new migration, alter `enforce_last_tree_membership()`'s function body to add one check before the existing count check: `if not exists (select 1 from individuals where id = old.individual_id) then return old; end if;` — i.e., if the individual row itself no longer exists (it is being deleted in the same statement/transaction), the "don't strip someone's last tree" guard is moot and must not block the cascade. This is a one-line, narrowly-scoped fix: the guard still fully protects the original single-tree-removal path (where the individual row still exists throughout), and now also correctly allows the cascade-on-full-delete path. No change to `enforce_no_relationships_before_membership_removal` or to `deleteIndividual()`'s call pattern is needed.

**Alternatives considered**:
- A dedicated `delete_individual_everywhere(individual_id uuid)` SECURITY-INVOKER RPC (mirroring `set_default_family_tree`'s shape from `0008_family_tree_functions.sql`) that disables/bypasses the guard triggers explicitly for this one operation: rejected — more moving parts (a new function, a new `grant execute`, a new client call site replacing the existing `.delete()` call) than a one-line trigger-function fix that keeps `deleteIndividual()`'s existing shape and call sites in `DeleteIndividualDialog.tsx` completely unchanged.
- Reordering `deleteIndividual()` to delete `individual_tree_memberships` rows itself, one at a time, before deleting the `individuals` row: rejected — this still runs into the same trigger logic (the individual row would still exist while its second-to-last membership is removed, so `enforce_last_tree_membership` would legitimately still fire on that row), so it doesn't actually solve the problem without the same guard-logic fix; it would also require deciding an artificial deletion order, adding complexity for no benefit over fixing the guard once.

## 5. "Already deleted by someone else" handling (FR-013)

**Finding**: `updateIndividual(id, input)` (`individualService.ts:52-62`) uses `.single()`, which surfaces Postgres/PostgREST code `PGRST116` ("no rows returned") when the row no longer exists — `toDataAccessError()` has no case for it today, so it falls through to a generic `"UNKNOWN"`/"Đã xảy ra lỗi. Vui lòng thử lại." message, not a clear "this record no longer exists" message. `deleteIndividual(id)`'s final `.delete().eq("id", id)` returns `{ error: null }` with zero rows affected when the id doesn't exist — no error at all, so calling delete on an already-deleted individual today silently "succeeds" with no feedback.

**Decision**: Add a `PGRST116` → `DataAccessError("NOT_FOUND", "Cá thể này không còn tồn tại (có thể đã bị người khác xoá).")` branch to `toDataAccessError()` in `individualService.ts`, which `updateIndividual` already benefits from automatically since it already routes its Postgres error through `toDataAccessError()`. For `deleteIndividual`, add an explicit pre-check: request the row's existence (or check the delete response's affected-row info) and throw the same `NOT_FOUND` error when nothing was deleted, so the dashboard can show "this individual no longer exists" instead of a false "success."

**Alternatives considered**: Relying on optimistic UI + a background refetch to "eventually" notice the row is gone (React Query's cache would naturally drop it on the next list refetch): rejected as insufficient on its own — spec 007's FR-013 explicitly requires the *action itself* (the specific edit/delete click) to be rejected with a clear message, not just a delayed list refresh.

## 6. Reusing `IndividualForm` and `DeleteIndividualDialog` outside the tree canvas (FR-007, FR-011)

**Finding**: `IndividualForm`'s `treeId` prop is read only in create-mode code paths (`createIndividual(treeId, ...)`, the "Liên kết với" relationship-linking section gated on `!initialIndividual`); in edit mode (`initialIndividual` set), `treeId` is never actually used — `updateIndividual(initialIndividual.id, input)` takes no tree argument. `DeleteIndividualDialog`'s underlying `deleteIndividual()` call, per §4, is already tree-independent (system-wide). Neither component's actions (`canManage`-gated in `TreeWorkspace.tsx`) depend on anything beyond `admin`/`editor` role and a selected individual.

**Decision**: Reuse both components unmodified in shape inside the new dashboard page: `IndividualForm` is rendered in a `Modal` with `treeId` satisfied by any placeholder string (dead in edit mode, confirmed by the trace above) and `existingIndividuals` omitted (defaults to `[]`, which also correctly suppresses the create-only relationship-linking section). `DeleteIndividualDialog` is reused as-is, with one copy change: its current confirmation text only mentions relationships being deleted, never family-tree memberships — add a sentence stating the individual will be removed from every family tree they belong to, satisfying FR-011 exactly (this is a text-only change to the existing component, used both here and in its original `TreeWorkspace.tsx` context, where the added sentence is equally true and not misleading).

**Alternatives considered**: Forking a dashboard-specific copy of each component to avoid a `treeId` placeholder prop: rejected — `IndividualForm`'s props already make `treeId` required only because create-mode needs it; introducing an optional variant (`treeId?: string`, required only when `!initialIndividual`) is a larger, riskier change to a shared component's type surface for a purely cosmetic prop-passing concern in the one new call site.

## 7. Access control (FR-012)

**Finding**: `RequireRole` already supports an arbitrary `allow: UserRole[]` list with zero code changes needed; today's two admin routes both use `allow={["admin"]}` only — no existing route combines `admin` + `editor`. `individuals_admin_editor_update`/`_delete` RLS policies (`0007_rls_policies.sql:71-78`) already permit both roles at the database layer, so no new RLS policy is needed for the dashboard's write actions.

**Decision**: Register the new route with `RequireRole allow={["admin", "editor"]}`, matching FR-012 and FR-009 (assumption: admin/editor have identical dashboard permissions) with no changes to `RequireRole` itself.

**Alternatives considered**: None needed — this is a direct, zero-risk application of an existing, already-general mechanism.
