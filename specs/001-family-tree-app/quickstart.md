# Quickstart: Bùi Family Genealogy Tree (Gia Phả)

This guide validates the feature end-to-end once implemented. It does not contain implementation code — see `data-model.md` for schema, `contracts/` for interfaces, and `tasks.md` (generated by `/speckit-tasks`) for build steps.

## Prerequisites

- Node.js 20 LTS and npm installed.
- A Supabase project (free tier is sufficient) with the schema from `supabase/migrations/` applied.
- A `.env` file at the repo root (git-ignored) containing:
  ```
  VITE_SUPABASE_URL=<your-project-url>
  VITE_SUPABASE_PUBLISHABLE_KEY=<your-project-publishable-key>
  ```
  (see committed `.env.example` for the exact variable names — never commit real values)
- Three Supabase Auth users seeded with `profiles.role` = `admin`, `editor`, and `viewer` respectively (for role-based validation below).

## Setup

```bash
npm install
npm run dev
```

Open the printed local URL in a browser.

## Validation scenarios

Each scenario maps to an acceptance scenario in `spec.md`. Run them in order; later scenarios assume earlier ones passed.

### 1. View the default tree (User Story 1)

1. Sign in as `viewer`.
2. **Expect**: the home page automatically shows the default family tree within ~3 seconds (SC-001), rendered as nodes/edges.
3. Collapse a branch at any individual with descendants. **Expect**: descendants disappear, layout adjusts.
4. Expand it again. **Expect**: they reappear.
5. Select any individual. **Expect**: a detail panel shows full name, alias, dates, gender, notes, avatar (if set).
6. Confirm parent-child, spouse, and sibling edges are visually distinguishable (e.g., by color/line style).
7. Use the "Export/Print" action. **Expect**: a PNG or PDF download reflecting exactly the current expand/collapse state (FR-027, SC-008).
8. Confirm no add/edit/delete/import/tree-management controls are visible or reachable as `viewer`.

### 2. Manage individuals and relationships (User Story 2)

1. Sign in as `editor`.
2. Create a new individual with only `FullName` and `Gender` set. **Expect**: saved successfully; not yet visible on the tree (no relationship yet).
3. Add a `parent_child` relationship linking the new individual to an existing one. **Expect**: appears on the tree immediately.
4. Edit the individual's `Notes` to exactly 100 characters. **Expect**: saves. Attempt 101 characters. **Expect**: blocked with a message (FR-007).
5. Upload an avatar photo, then upload a second one. **Expect**: the second replaces the first (FR-008).
6. Attempt to delete an individual who has the relationship from step 3. **Expect**: blocked or asks for explicit cascade confirmation (FR-012).
7. Remove the relationship, then delete the individual. **Expect**: succeeds.

### 3. Bulk import (User Story 3)

1. Sign in as `editor` or `admin`.
2. Prepare a sample `.xlsx` file following `contracts/xlsx-import-template.md`, including at least one row with a missing `Họ tên` and one row that exactly duplicates an existing individual's `Họ tên` + `Ngày sinh`.
3. Upload it to a chosen tree. **Expect**: a summary showing succeeded/failed/duplicate counts; the invalid row is reported with a reason; the duplicate row is flagged, not silently re-imported; valid rows (and the relationships derived from their `Mã cha`/`Mã mẹ`/`Mã vợ/chồng` columns) appear on the tree.
4. Upload a file with no sheets. **Expect**: rejected outright with an explanation (FR-015), no partial import.

### 4. Multiple trees and default selection (User Story 4)

1. Sign in as `admin`.
2. Create up to 5 trees total. **Expect**: succeeds each time.
3. Attempt a 6th. **Expect**: blocked, limit explained (FR-017).
4. Mark a non-default tree as default. **Expect**: home page (re-visit as any role) now shows that tree.
5. Attempt to delete the current default tree. **Expect**: blocked or requires assigning a new default first (FR-021).
6. Reduce to exactly one tree, then delete it. **Expect**: succeeds (FR-021); the home page shows the "no default tree" empty state.

### 5. Role enforcement is server-side, not just UI (security check)

1. While signed in as `viewer`, open the browser devtools console and attempt to call the Supabase client directly to insert an individual (bypassing the UI).
2. **Expect**: the request is rejected by the database (RLS `PERMISSION_DENIED`), confirming enforcement does not rely solely on hidden UI controls (FR-010, FR-011, FR-020; see `data-model.md` RLS table).

### 6. Guest (unauthenticated) viewing of a published tree (User Story 4 / FR-028)

1. Sign in as `admin`, open "Quản lý cây gia phả", and mark the default tree "Công khai" (published).
2. Open the home page in a private/incognito window (no session). **Expect**: the tree renders within ~3 seconds (SC-009), with expand/collapse, individual detail, and export/print all working exactly as for a signed-in Viewer.
3. Confirm no add/edit/delete/import/tree-management controls are visible, and the sidebar's "Tài khoản" section shows "Đăng nhập" instead of "Đăng xuất".
4. In the same guest session's devtools console, attempt to call the Supabase client directly to insert or update an individual. **Expect**: rejected by RLS (`PERMISSION_DENIED`) — publishing only ever grants read access (FR-029).
5. Sign back in as `admin` and mark the tree private again. **Expect**: reloading the home page in the guest window now shows a "please sign in" prompt instead of the tree.

## Success

All scenarios above passing, with the measurable targets from `spec.md` Success Criteria (SC-001–SC-009) met, indicates the feature is ready for acceptance review.
