**Feature Branch**: `002-lunar-events-tree-slugs`

**Created**: 2026-07-20

**Status**: Draft

**Input**: User description: "display lunar date (birthday and death date) in modal person detail. the date store in db is gregorian date, the web client need compute the correct lunar date based on the gregorian date to display. / in the left sidebar, add one more item to see nearly events, which show the calendar grid with event by month (birthday and date of death). the calendar grid should show both gregorian and lunar dates. highlight the dates have events. taps a date to see detail (which events for that date), show no event for this date if no event. admin can config send email to notify for a specific and nearly event (include template email, number of days to send email for next event - default is 7 days) / each family tree should have specific url to view the family tree, each faimily tree has a slug value, this value auto generated by the name of the family tree, creator (admin/editor) can modify it. ## Clarifications

### Session 2026-07-20

- Q: Who can create a family tree and edit its slug, given the existing app already restricts tree creation/deletion/publishing to Admin only? → A: Admin-only — no change to the existing permission model; Editors cannot create trees or edit slugs.
- Q: Should anonymous guests viewing a public family tree also see the new Upcoming Events calendar and lunar-date display? → A: Yes — same visibility as the rest of that public tree, consistent with the birth/death info already shown on public person profiles.
- Q: Should the admin-managed email-reminder settings (template, days-before, recipients) be one global configuration or per-family-tree? → A: The template and days-in-advance are a single, application-wide configuration; the recipient list defaults to an application-wide list but can be overridden for a specific family tree.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - See Lunar Dates Alongside Gregorian Dates (Priority: P1)

A family member viewing a person's detail wants to see the traditional lunar calendar date for that person's birthday and, if applicable, date of passing — not just the Gregorian (Western) date already stored — because lunar dates are what many family customs (birthday observances, death anniversaries) are actually based on.

**Why this priority**: This is the smallest, most self-contained piece of value and directly enables the more complex calendar feature (Story 2). It can ship alone and immediately improves every existing person record without any new data entry.

**Independent Test**: Open any person's detail view who has a known, complete birth date; confirm the lunar-calendar equivalent appears next to the Gregorian date. Can be fully tested without touching the calendar or tree-URL features.

**Acceptance Scenarios**:

1. **Given** a person with a fully specified birth date (day, month, and year all known), **When** a user opens that person's detail view, **Then** the view shows both the Gregorian birth date and its computed lunar-calendar equivalent.
2. **Given** a person marked as deceased with a fully specified date of death, **When** a user opens that person's detail view, **Then** the view shows both the Gregorian date of death and its computed lunar-calendar equivalent.
3. **Given** a person whose birth or death date is only partially known (e.g., year only, or unknown), **When** a user opens that person's detail view, **Then** the view shows the Gregorian information as it exists today and clearly indicates that a lunar date cannot be determined, without showing an incorrect or misleading lunar date.
4. **Given** a person with no death date recorded because they are not marked deceased, **When** a user opens that person's detail view, **Then** no lunar date is shown for death.

---

### User Story 2 - Browse Upcoming Life Events on a Calendar (Priority: P2)

A family member wants a dedicated place, reachable from the sidebar, to browse a month-by-month calendar that shows both Gregorian and lunar dates and highlights which days have a birthday or death anniversary, so they can plan ahead for observances without checking every person's profile individually. An administrator additionally wants to configure automatic email reminders so nobody misses an upcoming observance.

**Why this priority**: Builds directly on Story 1's date computation and delivers the main "don't miss an important date" value the user is asking for. It depends on Story 1 being available but is independently testable and deployable once it is.

**Independent Test**: From the sidebar, open the new "Upcoming Events" view, browse to a month containing a known birthday or anniversary, confirm the day is highlighted, tap it, and confirm the correct event(s) are listed. Can be tested without the email notification piece by simply verifying the calendar and detail views.

**Acceptance Scenarios**:

1. **Given** a user is viewing a family tree — whether signed in, or an unauthenticated guest viewing a public tree — **When** they select the "Upcoming Events" item in the left sidebar, **Then** a calendar grid for the current month opens, with every day showing both its Gregorian date and its lunar-calendar date.
2. **Given** the displayed month contains one or more days that are the recurring birthday or death anniversary of a person in the tree, **When** the calendar renders, **Then** those specific days are visually highlighted as having events.
3. **Given** a highlighted day, **When** a user taps/clicks it, **Then** a detail view lists every matching event for that day (person name and event type — birthday or death anniversary).
4. **Given** a day with no matching events, **When** a user taps/clicks it, **Then** the detail view clearly states there are no events on that date.
5. **Given** a user navigates to a different month, **When** the new month renders, **Then** its highlighted days and both calendar systems update accordingly.
6. **Given** an administrator opens event-notification settings, **When** they set the number of days-in-advance to send a reminder (defaulting to 7 if left unset) and provide/edit the email template, **Then** the settings are saved and used for future reminder emails.
7. **Given** notifications are enabled and an upcoming event falls within the configured reminder window, **When** the scheduled check runs, **Then** a reminder email is generated using the configured template, populated with that event's details.
8. **Given** an upcoming event is further away than the configured reminder window, **When** the scheduled check runs, **Then** no reminder email is sent yet for that event.

---

### User Story 3 - Reach a Family Tree by a Friendly, Sharable URL (Priority: P3)

An administrator — the only role permitted to create and manage family trees — wants each family tree to have a short, readable identifier (a "slug") derived from its name, so the tree can be shared and opened directly via its own URL, while the home screen continues to show only the one tree marked as the public, default tree.

**Why this priority**: Independently valuable (sharing/bookmarking a specific tree) but not required for Stories 1 or 2 to work, and it touches a different part of the app (tree management/routing) rather than person data or dates.

**Independent Test**: Create or open a family tree, confirm it has a generated slug, edit that slug, and confirm the tree can be opened by visiting its slug-based address — all without any dependency on lunar dates or the events calendar.

**Acceptance Scenarios**:

1. **Given** an administrator names a new family tree, **When** the tree is saved, **Then** the system automatically generates a unique, readable slug from the tree's name.
2. **Given** an existing tree, **When** an administrator edits its slug to a new value, **Then** the system validates it is unique and in a URL-safe format before saving, and rejects the change with a clear message if it collides with another tree's slug or is invalid.
3. **Given** the tree marked as both "default" and "public," **When** any visitor opens the application's home screen, **Then** only that tree is shown — never any other tree, regardless of the other trees' slugs.
4. **Given** a family tree that is not the default tree but is marked as viewable by its slug, **When** an authorized visitor opens that tree's slug URL, **Then** that specific tree loads correctly.
5. **Given** a family tree that is not viewable by slug (not made publishable), **When** anyone attempts to open its slug URL, **Then** the system denies access and shows an appropriate message instead of the tree's content.

---

### Edge Cases

- What happens when a person's birth or death date falls on a lunar leap month? The displayed lunar date must unambiguously indicate the leap month rather than silently mapping to the regular month.
- What happens when the Gregorian date falls outside the range the lunar conversion supports? The system must show the Gregorian date and clearly state the lunar equivalent is unavailable, rather than showing a wrong date.
- What happens when two people in the same tree share the same event day? The date-detail view must list all of them, not just one.
- What happens when an administrator disables event notifications entirely? No reminder emails are sent, and existing scheduled reminders are cancelled.
- What happens when a tree's name changes after its slug was already set or edited? The slug does not change automatically; only an explicit edit changes it, so existing shared links keep working.
- What happens when someone tries to set a family tree's slug to a value already used by another tree (including a soft-deleted or renamed one)? The change is rejected with a clear explanation.
- What happens when there is no family tree currently marked as "default and public"? The home screen shows a clear message instead of a blank or broken page.
- What happens when a family tree has no tree-specific recipient override configured? Its event reminders use the application-wide recipient list instead.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST compute a lunar-calendar date equivalent for any birth or death date that has full day-level precision, using the traditional lunar calendar as the basis for conversion.
- **FR-002**: System MUST display the computed lunar date alongside the existing Gregorian date in the person detail view, for both birthday and date of death.
- **FR-003**: System MUST NOT display a lunar date, and MUST instead indicate it is unavailable, when the underlying Gregorian date is not fully known (year-only, month-only, or unknown precision) or falls outside the supported conversion range.
- **FR-004**: System MUST add a new sidebar navigation item that opens an "Upcoming Events" calendar view.
- **FR-005**: The calendar view MUST display a month-based grid where each day shows both its Gregorian date and its corresponding lunar date.
- **FR-006**: The calendar view MUST visually distinguish days that have at least one associated birthday or death-anniversary event from days that do not.
- **FR-007**: Selecting a day in the calendar MUST show the list of matching events for that day (person and event type), or a clear "no events" message if none match.
- **FR-008**: Birthday and death-anniversary events MUST recur every year on the person's Gregorian birth/death month-and-day, independent of which year the calendar is currently displaying.
- **FR-009**: System MUST allow an administrator to configure, as a single application-wide setting, whether automatic email reminders for upcoming events are enabled or disabled.
- **FR-010**: System MUST allow an administrator to configure a single, application-wide email template used for event-reminder notifications.
- **FR-011**: System MUST allow an administrator to configure a single, application-wide number of days in advance of an event a reminder email is sent, defaulting to 7 days when not otherwise configured.
- **FR-011a**: System MUST allow an administrator to configure a default, application-wide list of recipient email addresses that receive event-reminder notifications.
- **FR-011b**: System MUST allow an administrator to override the recipient list for a specific family tree; events belonging to that tree use the override instead of the application-wide list, while all other trees continue using the application-wide list.
- **FR-012**: System MUST send a reminder email for each upcoming event that falls within the configured advance-notice window, using the configured template populated with that event's details, when notifications are enabled.
- **FR-013**: System MUST NOT send a reminder email for an event outside the configured advance-notice window, and MUST NOT send duplicate reminder emails for the same event occurrence.
- **FR-014**: System MUST generate a unique, URL-safe slug automatically from a family tree's name at the time the tree is created.
- **FR-015**: System MUST allow only an administrator (the only role permitted to create and manage family trees) to edit a tree's slug after creation, rejecting edits that are not unique or not URL-safe.
- **FR-016**: System MUST ensure the home screen displays only the family tree currently marked as the default, public tree, regardless of how many other trees exist.
- **FR-017**: System MUST allow a family tree that is marked as viewable-by-slug to be opened directly via its slug-based address, independent of whether it appears on the home screen.
- **FR-018**: System MUST deny access via slug URL to a family tree that is not marked as viewable-by-slug, and MUST show a clear message rather than exposing its content.
- **FR-019**: System MUST automatically assign a generated slug to every family tree that already exists at the time this feature is rolled out, so that no existing tree is left without one.
- **FR-020**: System MUST preserve all existing person and family-tree data unchanged when adding the new lunar-date, event-notification, and slug capabilities — the rollout MUST only add new information, never remove or alter existing birth dates, death dates, or tree records.
- **FR-021**: System MUST make the lunar-date display and the Upcoming Events calendar visible to unauthenticated guests viewing a public family tree, with the same visibility as that tree's existing person details — guests are not required to sign in to use either capability on a public tree.

### Key Entities

- **Person (existing entity, extended in behavior only)**: An individual within a family tree; already has a Gregorian birth date and, if deceased, a Gregorian death date, each with a precision level (full date, year-only, etc.). This feature adds a *computed* lunar-date equivalent wherever the underlying date has full precision — no new stored fields on the person are implied by lunar-date display.
- **Life Event**: A recurring, yearly occurrence derived from a person's birth date or death date — either a "birthday" or a "death anniversary." Carries the person it belongs to, the event type, and both its Gregorian and lunar date form for the current year.
- **Event Notification Configuration**: Administrator-managed settings controlling automatic reminder emails, scoped as a single application-wide configuration: whether reminders are enabled, the email template content, and the number of days in advance to send a reminder (default 7). Its recipient list defaults application-wide but MAY be overridden per family tree.
- **Family Tree (existing entity, extended)**: A collection of people and relationships; already has a name, a default flag, and a public/private visibility setting. This feature adds a unique, editable **slug** (administrator-managed only) used to build a shareable URL for that specific tree, and an optional recipient-list override for event-reminder emails.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: A user viewing any person with a fully known birth or death date can see the correct lunar-calendar equivalent without navigating away from the person's detail view.
- **SC-002**: 100% of life events with a fully known Gregorian month and day are correctly highlighted on their corresponding day when a user browses to that month in the events calendar.
- **SC-003**: A user can go from the sidebar to seeing the full list of events for any chosen date (or a clear "no events" message) in two actions or fewer.
- **SC-004**: An administrator can update the reminder lead time and email template and have it take effect for the next scheduled check without any technical assistance.
- **SC-005**: Every eligible upcoming event within the configured reminder window results in exactly one reminder email per occurrence — no misses, no duplicates — when checked over a full monitoring cycle.
- **SC-006**: 100% of family trees, both pre-existing and newly created, have a valid, unique slug immediately after this feature is rolled out.
- **SC-007**: A person with a slug-accessible tree's URL can open that specific tree directly, without it ever needing to appear on the home screen.
- **SC-008**: The home screen continues to show exactly one family tree (the default, public one) both before and after this feature is rolled out, with zero change to that existing behavior.
- **SC-009**: An unauthenticated guest viewing a public family tree sees the same lunar dates and Upcoming Events calendar as a signed-in user would, with no additional sign-in step required.

## Assumptions

- The lunar calendar used for conversion is the traditional East Asian lunar calendar (the same one used for Vietnamese lunar birthdays and death-anniversary observances), consistent with this application's purpose as a Vietnamese family-tree ("Gia Phả") tool.
- "Publishable" (as used for viewing a non-default tree by slug) is the same visibility setting already used to mark a tree as public, rather than a new, separate flag.
- The existing partial-date precision levels (full date, year-only, month-only, unknown) remain unchanged; this feature only adds a computed lunar view on top of dates that are already fully known, and never invents a lunar date for a partially known one.
- Reminder emails are sent on a recurring, automated schedule (e.g., a daily check); the exact scheduling mechanism is an implementation decision outside the scope of this specification.
- Existing family trees and person records require a one-time, automatic backfill (new slug values for trees; default notification settings) as part of rollout, with no manual data re-entry and no disruption to existing users.
- Slug generation transliterates non-ASCII characters (such as Vietnamese diacritics) into a URL-safe form and appends a distinguishing suffix on name collisions to keep every slug unique.
