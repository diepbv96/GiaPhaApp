# Phase 0 Research: Lunar Dates, Upcoming Events Calendar & Shareable Tree URLs

No `NEEDS CLARIFICATION` markers remain in the Technical Context — the three genuinely open decisions below (lunar algorithm, email delivery, slug generation) had no explicit answer from the project owner but do have a clear, low-risk default for this app's scale, so they are recorded here as researched decisions rather than sent back for clarification.

## 1. Gregorian → lunar date conversion

- **Decision**: Implement the well-known Hồ Ngọc Đức astronomical algorithm for the Vietnamese lunar calendar directly as a small, dependency-free, pure TypeScript module (`src/lib/lunarCalendar.ts`), computed relative to the UTC+7 (Vietnam) time zone, returning `{ day, month, year, isLeapMonth }` for any Gregorian calendar date within the algorithm's high-accuracy range (1800–2199), and `null` for anything outside that range or not applicable.
- **Rationale**: This is the same algorithm underlying essentially every Vietnamese lunar-calendar tool and website; it is public-domain, well-documented, and requires no external service call or paid API — appropriate for a small family app that must keep working offline-of-third-parties and match the "Vietnamese lunar calendar" assumption already recorded in spec.md. Because it is pure math (no timezone-sensitive `Date` object mutation beyond the initial UTC+7 anchor), it is trivial to unit test with fixed reference-date pairs (e.g., known Tết dates) and needs no new runtime dependency.
- **Alternatives considered**: An npm lunar-calendar package (e.g. `lunar-javascript`, `vietnamese-lunar-calendar`) — rejected; several are unmaintained or bundle far more (Chinese zodiac, fortune-telling extras, multi-locale) than this feature needs, and the core conversion math is short enough to own directly and test precisely. A server-side/database conversion (Postgres function) — rejected; the spec explicitly states "the web client need compute the correct lunar date," and keeping it client-side avoids a round trip for a value that's pure math.

## 2. Scheduled email reminders

- **Decision**: A single new Supabase Edge Function (`supabase/functions/send-event-reminders`), triggered once daily by Supabase's built-in Cron Trigger for Edge Functions (no extra infrastructure to provision). The function reads the singleton `event_notification_config` row, computes which upcoming Life Events fall exactly at the configured days-before window across all trees' individuals, resolves each affected tree's recipient list (per-tree override if present, else the global list), skips any event occurrence already present in `event_notification_log`, sends one email per occurrence via a plain `fetch` call to a transactional email API (Resend — simple single-endpoint REST API, generous free tier, no SDK needed), and records each send in `event_notification_log` to guarantee no duplicates (FR-013). The due-event/dedupe decision logic is written as a small, framework-agnostic module (`logic.ts`) with no Deno-only or Node-only APIs, so it can be unit-tested with Vitest exactly like the rest of the codebase, and the Deno `index.ts` handler is a thin wrapper around it.
- **Rationale**: The app has no existing backend server (001's explicit architecture decision), so a scheduled job of any kind requires *some* new server-side execution surface — a single-purpose Edge Function is the smallest addition that fits Supabase-as-the-only-backend, mirrors how Supabase's own docs recommend building exactly this "scheduled digest/reminder email" pattern, and keeps the addition isolated (one function, one job) rather than growing into a general API. Resend is chosen as a concrete default because the plan needs one to be buildable and testable; it is a widely used, minimal-integration choice with no SDK dependency (a single authenticated POST request), but the function's `fetch`-based design means swapping providers later is a one-file change.
- **Alternatives considered**: A third-party workflow tool (e.g., a cron SaaS calling a webhook) — rejected, unnecessary extra moving part when Supabase already provides Cron Triggers for Edge Functions. Sending via Supabase Auth's built-in email — rejected, that pathway is reserved for auth-flow emails (confirmation/reset) and is not designed for arbitrary templated business notifications. A heavier email SDK/library — rejected as unneeded ceremony for a single POST-per-email volume this small (spec Scale/Scope: a few thousand person-events per run, in practice far fewer actual sends).

## 3. Family tree slug generation

- **Decision**: A small, dependency-free pure function (`src/lib/slug.ts`) that: Unicode-normalizes (NFD) the tree name, strips diacritics, special-cases the Vietnamese `đ`/`Đ` (which NFD does not decompose) to `d`, lowercases, replaces runs of non-alphanumeric characters with a single hyphen, and trims leading/trailing hyphens. Uniqueness (including the collision-suffix, e.g. `-2`, `-3`, ...) is resolved with one round-trip query against `family_trees` at generation/edit time, enforced authoritatively by a `UNIQUE` constraint in the database (client-side generation is a convenience/UX layer, not the source of truth).
- **Rationale**: Vietnamese diacritic-to-ASCII transliteration is a narrow, well-understood string problem; existing generic `slugify` npm packages either mishandle `đ` (treating it as a plain Latin `d` already, or leaving it as a combining-mark artifact) or require a locale plugin for correct Vietnamese behavior, so hand-writing the ~15 lines needed is both more correct and avoids an extra dependency for a one-file utility, matching the project's existing lean-dependency style (research.md §1). Enforcing uniqueness in the database (not just in-app) is required regardless of how the slug string itself is produced, since it is also relied upon as the routing key (FR-014/FR-017/FR-018).
- **Alternatives considered**: `slugify` (npm) with a custom Vietnamese character map passed in — considered viable but rejected in favor of the fully self-contained version, since the custom-map option still needs the same `đ` special-casing to be supplied by hand, making the dependency add no real value here. Letting slugs be freely assigned by the creator with no auto-generation — rejected; spec FR-014 explicitly requires auto-generation from the name, with editing as a secondary, optional step.

## 4. Calendar grid UI (dual Gregorian/lunar month view)

- **Decision**: A hand-rolled month-grid component (`src/features/events/CalendarGrid.tsx`) built the same way as the rest of the UI (React + Tailwind, no new UI-kit dependency), rendering a 6×7 day grid where each cell shows its Gregorian day number plus the lunar day/month computed via `lunarCalendar.ts`, and a highlight state derived by matching each visible day against the currently-open tree's individuals' birth/death month-and-day (already fetched for the tree view — no new query pattern, just a client-side derivation hook, `useMonthEvents.ts`).
- **Rationale**: Off-the-shelf React calendar/date-picker libraries are built around a single calendar system and a "pick one date" interaction; none natively support rendering two calendar systems per cell or a per-day multi-event highlight/detail interaction, so adapting one would fight its abstractions more than a focused, purpose-built grid component costs to write — consistent with 001's precedent of hand-rolling the tree canvas layout rather than forcing a generic library to do bespoke work.
- **Alternatives considered**: `react-day-picker` / `@fullcalendar/react` — rejected for the reason above; either would need significant custom cell-rendering overrides to show a second calendar system, at which point most of the library's value (built-in single-date interactions) is unused.
